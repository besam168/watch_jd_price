- 今天在 QQ 私聊场景下，以“项目经理 / 技术总控 / 最终验收”模式，开始真实调度 Codex CLI 推进 `tmall-genie-voice-bridge`。

- 关键执行经验：不要一上来反复折腾复杂的一次性 shell 拼接；在 Windows + PowerShell 环境里，长 prompt、重定向、PATH 继承都容易出坑。
- 本次踩到的 3 类典型问题：
  1. PowerShell 的 `<` 重定向写法与预期不符，外层命令直接报错；
  2. 长 prompt 在 shell / 子进程链路中被拆参，导致 `audio` 等片段被 Codex 当成意外参数；
  3. 后台 Python 子进程未继承到 `codex` PATH，导致 `FileNotFoundError`。
- 当前更稳的调度策略：
  1. 先单独验证 Codex CLI 本体可用；
  2. 尽量在临时副本目录中执行，避免直接污染原项目；
  3. 若已成功进入 Codex 交互界面，优先直接向现成会话粘贴英文任务，而不是继续在外层折腾复杂启动参数；
  4. 需要脚本调用时，优先使用可执行文件绝对路径；
  5. 项目经理负责控边界、控验收口径，严禁执行层 AI 夸大“已完成/已验证”的能力。
- 对 `tmall-genie-voice-bridge` 这类项目，当前真实可复用方法论是：
  - 我负责中文沟通、任务拆解、阶段验收；
