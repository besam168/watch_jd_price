# Self Evolution 项目落地教程（可迁移到别的电脑）

> 目标：把“自我进化”做成一套**能长期自动推进、能和实际工作结合、能迁移到其他电脑/其他 agent** 的工作流，而不是一次性笔记。

---

# 一、这套方案到底是干什么的

这不是普通学习区，也不是随手收藏夹。

这套 `self-evolution` 项目的目标是：

- 持续从外部世界获取新信号
- 把新信号转成真实能力
- 优化实际工作流
- 通过每天一点点改进，长期积累成更强的执行能力

一句话原则：

**真实工作里发现卡点 → 当场解决 → 沉淀成脚本 / skill / 模板 / 工作流 → 下次更稳、更快、更强。**

也就是说：
- 不是为了学而学
- 不是为了记而记
- 而是围着真实工作中的问题进化

---

# 二、适用场景

这套方案适合：

1. 你有一个长期运行的 AI 助手 / Agent
2. 你希望它不是被动回答，而是能持续变强
3. 你希望它学来的东西能回灌到日常工作里
4. 你希望它每天/每周都有轻量推进，而不是全靠临时想起
5. 你未来可能会把同一套工作法迁移到别的电脑、别的账号、别的 agent

---

# 三、核心设计思想

## 1）自我进化必须和实际工作绑定
不是抽象成长，而是围绕：
- 工作中的卡点
- 工具不稳定
- 搜索/抓取失败
- 自动化链路不稳
- skill 不够用
- 工作流不顺手

只要这些问题出现，就把它当成“进化入口”。

---

## 2）双引擎信息源
这套方案不是只盯 GitHub。

而是：

### Reddit 负责提题
看：
- 用户在热议什么
- 真实抱怨点在哪里
- 哪些新方向开始升温
- 哪些需求不是技术圈自嗨，而是真有人在乎

### GitHub 负责解题
看：
- 谁把这件事做成了
- 真实项目怎么组织
- 文档怎么写
- 工作流怎么落地
- 哪些 repo 值得借鉴结构与方法

一句话：

**Reddit 提题，GitHub 解题。**

---

## 3）轻自动优先，不一开始就上重调度
先让 Agent 通过 heartbeat / 轻触发机制自然推进。

只有当你发现：
- 频率不稳
- 周报容易漏
- 某些动作必须固定时间跑

再加第二层硬调度（Windows 任务计划程序 / cron / 定时任务）。

这比一上来搞很重的自动化更稳。

---

# 四、目录结构建议

建议在工作区建立一个独立目录，例如：

```text
self-evolution/
  README.md
  github-learning-radar.md
  reddit-radar.md
  watchlist.md
  execution-checklist.md
  learning-log.md
  first-real-patrol-v1.md
  patrol-summary-YYYY-MM-DD.md
  scripts/
    run_real_patrol.py
```

如果是 OpenClaw 风格工作区，还建议在根目录配：

```text
HEARTBEAT.md
memory/
  YYYY-MM-DD.md
```

# 五、每个文件负责什么

## 1. `self-evolution/README.md`
这是总纲。

必须写清楚：
- 这套项目是干什么的
- 总原则是什么
- 节奏怎么跑
- 自动执行口径是什么
- 信息源是哪些

建议至少包括这些模块：
- 项目目标
- 总原则
- 当前双引擎阵地
- 当前主目标
- 执行节奏
- 自动执行说明
- 文件说明
- 工作原则

---

## 2. `github-learning-radar.md`
记录 GitHub 侧的长期观察方向。

建议写：
- 重点方向
- 候选 repo 类型
- 关键词
- 哪类项目值得深读
- 哪类项目只是噪音

例如方向可以包括：
- agent
- tool calling
- browser use
- computer use
- scraping / crawling
- memory / retrieval / RAG
- workflow / orchestration
- report automation
- notification / alerting
- dashboard / knowledge base

---

## 3. `reddit-radar.md`
记录 Reddit 侧要看的社区板块与热点模板。

建议至少写：
- 关注哪些 subreddit
- 看哪些类型的信号
- 热点记录模板
- 哪些热度值得追，哪些只是噪音

常见板块示例：
- `r/LocalLLaMA`
- `r/PromptEngineering`
- `r/OpenAI`
- `r/ChatGPT`
- `r/MachineLearning`
- `r/artificial`
- `r/singularity`

---

## 4. `watchlist.md`
这是长期候选池。

记录：
- 方向
- 关键词
- 候选 repo 类型
- 候选能力缺口
- 候选要补的工作流

重点不是记录“所有东西”，而是记录：
**未来最可能对实际工作有帮助的那些方向。**

---

## 5. `execution-checklist.md`
这是执行动作清单。

它的作用是防止 Agent 只会“看”，不会“做”。

建议里面分成：
- 轻巡逻动作
- 深读动作
- 周报动作
- 回灌动作

这样以后别的电脑上的 agent，拿到文件就知道每一步怎么推进。

---

## 6. `learning-log.md`
这是最关键的沉淀文件。

建议至少分成三个区域：

### `## 深读记录`
每次正式拆一个项目，就写在这里。

模板建议：
- 日期
- Repo
- 方向
- 解决的问题
- 最值得抄的 3 个点
- 文档/结构亮点
- 风险/边界
- 可迁移方法
- 回灌动作

### `## Heartbeat 推进记录`
每次 heartbeat 有实质推进，就写这里。

模板建议：
- 日期
- 本次 heartbeat 做了什么
- 为什么做这件事
- 解决了什么问题 / 捕捉到什么信号
- 沉淀到哪里
- 下次接着做什么

### `## 周报记录`
每周做一次中文收口。

写：
- 本周 Reddit 上开始火的方向
- 本周值得看的 GitHub 项目 / 方向
- 本周最值得实际开干的升级点
- 下周优先级

---

## 7. `HEARTBEAT.md`
这是轻自动的驱动层。

如果你的平台支持 heartbeat / 主动轮询 / 定期触发，建议把 heartbeat 任务写在这里。

推荐写法：
- 先读 `self-evolution/README.md`
- 再读 `learning-log.md`、`watchlist.md`、当天 memory
- 每次 heartbeat 只做一件最有价值的小事
- 优先级固定为：
  1. 先补真实工作里的能力缺口
  2. 再做轻巡逻
  3. 再推进深读/中文总结
- 没有实质进展就回复 `HEARTBEAT_OK`

这样 heartbeat 就不会变成垃圾输出机。

---

# 六、执行节奏怎么安排

## 每天：轻巡逻
目标：看有没有值得追的新信号。

动作：
- 快速看 Reddit 是否有持续热议的新方向
- 快速看 GitHub 是否有值得纳入候选池的新 repo
- 只记录最值得记的 1~3 个点

注意：
- 不强行产出
- 没有真信号就不写废话

---

## 每 2~3 天：深读 1 个项目
目标：从“看见”推进到“学会”。

动作：
- 选一个最值得的 repo
- 看它解决什么问题
- 看它结构怎么组织
- 看它工作流怎么设计
- 看它边界怎么表达
- 看它哪些方法值得迁移

产出：
- 一句话结论
- 3 个最值得抄的方法
- 1 个风险提醒
- 1 个回灌动作

---

## 每周：中文简报
目标：做决策收口，而不是堆信息。

内容包括：
- 本周 Reddit 最值得追的方向
- 本周 GitHub 最值得继续深读的项目
- 本周最值得实际落地的升级点
- 哪些只是噪音
- 下周优先级

---

## 遇到真实工作卡点：允许插队
这是整套方案最重要的一条。

只要真实工作里出现：
- 搜索不稳
- 抓取不稳
- 某个 skill 不够用
- 自动化链路不稳定
- 文档/边界表达太差

就允许直接插队。

默认优先级：

**真实工作里的痛点 > 学习节奏本身**

因为真正的进化，不是按计划表做完动作，而是把痛点变成能力。

---

# 七、首次落地步骤（在别的电脑上怎么搭）

如果你想把这套方案迁移到另一台电脑，建议按这个顺序来：

## 步骤 1：建立目录
创建：
- `self-evolution/`
- `self-evolution/scripts/`
- `memory/`
- 根目录 `HEARTBEAT.md`

---

## 步骤 2：先写总纲
先把 `self-evolution/README.md` 写好。

不要一开始就急着塞很多项目链接。

先把以下内容写清楚：
- 总原则
- 双引擎
- 节奏
- 自动执行说明
- 文件分工

没有总纲，后面容易越跑越乱。

---

## 步骤 3：补观察面
再写：
- `github-learning-radar.md`
- `reddit-radar.md`
- `watchlist.md`
- `execution-checklist.md`

这一步的目标是：
让 agent 知道去哪看、看什么、怎么判断值不值得追。

---

## 步骤 4：补沉淀面
创建 `learning-log.md`，把三个区先搭出来：
- 深读记录
- Heartbeat 推进记录
- 周报记录

先把模板写好。

这样它后面每次推进都有地方记。

---

## 步骤 5：配置轻自动
把 `HEARTBEAT.md` 改成真实执行清单，不要留空。

核心原则：
- 每次 heartbeat 只做一件最有价值的小事
- 没有实质进展时回复 `HEARTBEAT_OK`
- 有推进就落到 `learning-log.md` 和当天 memory

---

## 步骤 6：跑第一轮真实巡逻
不要只写模板。

第一轮一定要真实执行一次：
- 抓 Reddit 样本
- 抓 GitHub 样本
- 产出一个热点、一个优先深读项目、一个可回灌点

这样整套系统才从“文档”变成“活的工作流”。

---

## 步骤 7：把第一次结果写入日志和记忆
至少写到：
- `learning-log.md`
- `memory/YYYY-MM-DD.md`

如果你有 git，建议马上提交一次。

因为第一次巡逻结果，是以后验证这套系统有没有真的开始运转的重要节点。

---

# 八、推荐的自动执行口径

以后在别的电脑上，如果你让另一个 agent 接手，建议直接给它这段口径：

> 你要把 `self-evolution` 当成一个长期工作区，而不是普通学习笔记。
> 核心原则是：真实工作里发现卡点 → 当场解决 → 沉淀成脚本 / skill / 模板 / 工作流 → 下次更稳、更快、更强。
> 默认节奏：每天轻巡逻、每 2~3 天深读 1 个项目、每周给一份中文简报、遇到真实工作痛点允许插队。
> 默认用 heartbeat 做轻自动驱动；若后续频率或稳定性不足，再补硬调度。
> 每次推进都要落到 `learning-log.md` 和当天 `memory/YYYY-MM-DD.md`，避免空转。

这段话基本就能让另一个 agent 按同样的逻辑起步。

---

# 九、什么叫“这套方案跑起来了”

满足下面这些条件，就说明它已经不是摆设：

1. 已有固定目录结构
2. 已有 README 总纲
3. 已有轻巡逻与深读模板
4. 已有 heartbeat 驱动逻辑
5. 已跑过至少一次真实巡逻
6. 已产生第一批真实总结
7. 已开始把结果回灌到技能、脚本、模板或工作流

如果只停在“写了一堆文档”，那不算跑起来。

# 十、后续可升级项

当这套方案运行一阵之后，可以继续升级：

## 升级 1：增加 `next-actions.md`
专门放：
- 当前下一步
- 最高优先级
- 卡住点
- 等待验证项

## 升级 2：增加自动巡逻脚本
例如：
- Reddit RSS 抓取脚本
- GitHub topic 抓取脚本
- 巡逻结果汇总脚本

## 升级 3：增加硬调度
例如：
- Windows 任务计划程序
- cron
- 周报固定时间生成

## 升级 4：增加“回灌清单”
把学到的东西明确映射到：
- 哪个 skill 要升级
- 哪个脚本要补
- 哪个流程要重构
- 哪个文档要改写

---

# 十一、给未来自己的提醒

不要把 `self-evolution` 变成：
- 收藏夹
- 热点摘抄
- 术语仓库
- 假勤奋记录本

它必须始终围绕一件事：

**让 agent 的实际工作能力越来越强。**

如果一个学习动作不能改善：
- 速度
- 稳定性
- 判断力
- 工作流
- 可复用性

那它就不该占太多精力。

---

# 十二、一句话总结

这套方案的本质不是“学习系统”，而是：

**一个围绕真实工作持续升级的 Agent 进化工作流。**

你以后在别的电脑上，只要把目录、README、HEARTBEAT、learning-log 这些核心骨架搭起来，再让那个 agent 按这份教程执行，它就能复刻这套方案。
