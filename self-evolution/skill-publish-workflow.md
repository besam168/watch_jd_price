# Skill Publish Workflow

> 用途：当某个第三方 skill / 插件已经在本地工作区完成改造，下一步需要独立上传到 GitHub 时，避免把总工作区、嵌套 `.git`、无关脏文件或临时产物一起带上去。

---

## 适用场景

当出现以下情况时，优先走这份 workflow：
- 从第三方仓库下载了一个 skill，准备改成 OpenClaw 版本再发布
- 在总工作区 `skills/<name>` 下孵化了一个 skill，但要推到**独立 GitHub 仓库**
- 某个目录曾经自带 `.git`，容易被主工作区误识别成嵌套仓库 / submodule
- 当前总工作区本身很脏，不适合直接从根目录做发布

一句话原则：

**发布 skill 时，永远以 skill 目录为边界，不以总工作区为边界。**

---

## 默认发布顺序（写死）

### 1. 先把 skill 收成独立发布单元
检查目标目录只保留真正要发布的内容，例如：
- `SKILL.md`
- `LICENSE`
- `references/`
- 必要脚本 / 资源

删掉不必要的演示型、聊天型、截图型、仓库导流型文件，例如：
- demo 对话
- 二维码图
- 纯展示 README（如不准备保留）
- 临时测试产物

### 2. 检查目录里是否残留 `.git`
如果这个 skill 是从别的仓库 clone 下来的，优先检查：
- `<skill>/.git` 是否存在

如果存在，而且当前目标是把它作为**普通目录**纳入工作区或重新发布，就要先处理掉旧 `.git`，否则容易出现：
- 被主仓库误加成 embedded repo / submodule
- `git add` 后只记录一个 160000 指针
- 后续 push 到 GitHub 时内容不完整

### 3. 不要从总工作区直接推独立仓库
如果总工作区有大量无关修改，不要图省事直接在根目录操作远端。

默认做法：
- 进入 skill 目录
- 在该目录内重新初始化独立仓库，或拷贝到一个干净临时目录再初始化
- 只提交当前 skill 所需文件

### 4. 先本地验证发布内容边界
至少检查一次：
- `git status --short`
- `git ls-files`
- 目录树是否只包含预期文件

如果看到工作区根目录大量脏文件、上级目录变更、或无关脚本混进来，说明边界错了，先停。

### 5. 再接 GitHub 远端
确认目标仓库名无误后，再执行：
- `git remote add origin <repo>`
- `git branch -M main`
- `git push -u origin main`

优先使用 SSH remote。

## 推荐实操路线

### 路线 A：skill 目录内重建独立仓库（默认推荐）
适合：目标目录已经整理干净，且不想保留原第三方仓库历史。

步骤：
1. 确保 skill 目录内容已收口
2. 删除旧 `.git`（如果有）
3. 在 skill 目录下执行：
   - `git init`
   - `git add .`
   - `git commit -m "Initial OpenClaw skill release"`
4. 绑定目标 GitHub 仓库
5. push

优点：
- 最干净
- 不会把总工作区历史和脏状态带上去
- 最适合“第三方改造后再发布”

### 路线 B：复制到临时干净目录再发布
适合：当前 skill 目录还在总工作区里，担心误操作。

步骤：
1. 复制 `skills/<name>` 到临时目录
2. 在临时目录删掉无关文件 / `.git`
3. 在临时目录 `git init`
4. commit + push
5. 验证远端后，再决定是否同步回主工作区

优点：
- 风险最低
- 最不容易误推杂物

## 常见坑

### 坑 1：`git add` 后出现 embedded repository 警告
说明目录里还带着旧 `.git`。

处理：
- 先 `git rm --cached <dir>`
- 删除 `<dir>/.git`
- 再重新 add

### 坑 2：从总工作区直接看 `git status`，被海量无关修改干扰
根因不是 skill 本身有问题，而是发布边界错了。

处理：
- 不要继续在总仓库里硬推
- 改到 skill 目录或临时目录里独立发布

### 坑 3：README、图片、示例对话等展示文件混进正式发布
这类文件不一定错，但会让 skill 包变重、边界变虚。

处理：
- 发布前先决定：这是“展示仓库”还是“纯 OpenClaw skill 仓库”
- 如果目标是纯 skill，优先只保留运行与参考必需内容

### 坑 4：把“工作区内可用”误当成“可独立发布”
工作区里能用，不代表已经适合单独发布。

发布前至少回答 3 个问题：
1. skill 的边界文件是否清楚
2. 是否还带着外部仓库历史 / `.git`
3. 是否能在独立目录中完整提交并 push

---

## 最小发布检查表

发布前，至少过一遍：

- [ ] skill 目录已收口，只剩必要文件
- [ ] 已检查并处理旧 `.git`
- [ ] 未把总工作区当发布边界
- [ ] 已本地检查 `git status --short`
- [ ] 已确认远端仓库名 / 地址
- [ ] 默认优先 SSH remote

---

## 一句话总结

**第三方 skill 改造后再发布，最稳做法不是“在总工作区里顺手推”，而是“把 skill 收成独立发布单元，再单独 init / commit / push”。**
