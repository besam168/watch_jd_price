# self-evolution

## 项目定位

`self-evolution` 不是一个给终端用户直接使用的成熟产品，也不是一个已经工程化完成的情报平台。

它当前更准确的定位是：

**一个与实际工作绑定的“自我进化工作流原型”。**

它的核心不是为了刷概念，也不是为了看热闹，而是：

**在真实工作里发现卡点 → 当场解决 → 沉淀成脚本 / skill / 模板 / 工作流 → 下次更稳、更快、更强。**

Reddit / GitHub 双引擎巡逻，只是这套自我进化路线里的一个重要输入源，而不是全部。

一句话概括：

**真实工作里补能力缺口，外部巡逻里找新方法，最后都回灌到自己的工作流和执行体系。**

---

## 这个项目有什么用

它的作用不是“自动帮用户完成某个业务”，也不只是帮助操作者建立外部学习与情报巡逻流程。

它更重要的用途，是让自我进化**直接服务实际工作**。

也就是说：
- 不是刷概念
- 不是看热闹
- 不是为了显得自己在成长
- 而是直接提升干活能力

当前主要用途：

### 1. 把真实工作里的卡点，变成能力升级入口
真实工作中只要暴露出短板，例如：
- 搜索不稳
- 抓取不稳
- 某个 skill 卡点
- 自动化链路不稳
- 执行步骤太靠临场运气

就不能只临时绕过去，而要继续补成：
- 一个更稳的备用路线
- 一个新脚本
- 一个新 skill
- 一个更稳的执行顺序
- 一个可复用模板

### 2. 持续发现值得追的方向
不是漫无目的刷网页，而是围绕固定主题持续观察：
- AI agent framework
- workflow / automation
- memory / RAG / retrieval
- browser automation
- desktop automation
- prompt / persona / template
- report / monitor / summary pipeline

### 3. 把社区热点和真实实现连起来
很多外部热点只停留在讨论层；这套方案的目标是：
- 先从 Reddit 找信号
- 再去 GitHub 找实现
- 最后筛出真正值得学习和迁移的方法

### 4. 为当前项目提供方法升级输入
它适合给以下类型工作提供外部输入：
- OpenClaw skill 设计
- agent orchestration
- 自动化采集与报告流水线
- memory / retrieval 方案
- browser / desktop automation
- prompt / template 系统设计

## 默认路线（写死）

以后默认把 `self-evolution` 理解成下面这条固定路线：

**真实工作里发现卡点 → 当场解决 → 沉淀成脚本 / skill / 模板 / 工作流 → 下次更稳、更快、更强。**

这条路线是默认主线，不需要每次重新解释。

---

## 4 条硬原则

### 1. 自我进化必须服务实际工作
- 不是刷概念
- 不是看热闹
- 是直接提升干活能力

### 2. 每次进化都要落到工作流
目标不是“知道了更多”，而是：
- 更稳
- 更快
- 更少出错
- 更少依赖临场运气

### 3. 小改进也要记账
以下这些都算真实进化：
- 一次小补丁
- 一个新脚本
- 一个备用路径
- 一个新 skill
- 一个更稳的执行顺序

这些小东西长期积累起来，才是真正变强。

### 4. 遇到问题不能只绕过去，还要补能力缺口
如果真实工作里发现：
- Firecrawl 不稳
- 内置搜索不稳
- 抓取不稳
- 某个执行链路反复卡

就不能只说“今天先这样”，而要继续补成：
- 备用抓取路线
- 本机脚本能力
- 可复用巡逻脚本
- 必要时新增 skill

## 工作卡点入口（新增）

为了让“真实工作里发现卡点 → 当场解决 → 沉淀成能力资产”这条主线真正落地，当前新增两个固定入口：

### `worklog.md`
作用：
- 记录真实工作中暴露出来的卡点
- 记录这次是怎么先做完的
- 记录后续应该沉淀成什么能力资产
- 避免“今天解决了，明天又忘了”

### `bottleneck-template.md`
作用：
- 作为每次记录工作卡点的固定模板
- 强制把“问题现象 / 根因判断 / 沉淀动作 / 下次更稳的做法”写清楚
- 避免只写成流水账或情绪记录

### `worklog-radar-loop.md`
作用：
- 定义当工作卡点需要外部输入时，如何从 `worklog` 进入 `watchlist` 和 `self-evolution-radar`
- 固定“先记卡点，再跑巡逻，再回写结论”的联动顺序
- 避免把 radar 用成脱离实际工作的独立逛街动作

### `fallback-search-and-crawl-workflow.md`
作用：
- 定义搜索 / 抓取不稳时的固定 fallback 顺序
- 避免每次都靠临场记忆切换路线
- 把“先完成任务，再沉淀能力”的执行顺序写清楚

以后只要真实工作里出现：
- 搜索不稳
- 抓取不稳
- skill 卡点
- 自动化链路不稳
- 某一步太依赖临场运气

就应该优先进这两个文件，而不是只在聊天里说一句“下次注意”。

### 方法文档

#### `first-real-patrol-v1.md`
第一轮真实巡逻方案总说明。

作用：
- 定义第一轮巡逻目标
- 明确“先 Reddit、后 GitHub、再回灌”的总工作流
- 规定第一轮巡逻范围
- 给出最小输出模板和禁区

#### `execution-checklist.md`
执行节奏与交付清单。

作用：
- 定义“日常轻巡逻 / 深读 / 周报”三层节奏
- 规定每层该看什么、记什么、输出什么
- 避免陷入无效收藏和无边界深挖

#### `watchlist.md`
观察方向清单。

作用：
- 明确当前重点关注的技术方向
- 划定一级、二级关注范围
- 统一后续巡逻的雷达边界

#### `reddit-radar.md`
Reddit 侧的角色定义文件。

作用：
- 明确 Reddit 负责“热点发现、情绪观察、痛点提取、方向预警”
- 规定优先关注哪些 subreddit
- 说明每次巡逻需要记录什么

#### `github-learning-radar.md`
GitHub 侧的角色定义文件。

作用：
- 明确 GitHub 负责“验证实现、学习结构、吸收方法”
- 规定筛 repo 的标准
- 规定需要重点观察 README、目录结构、issue/discussions、维护状态等

---

### 执行脚本

#### `scripts/run_real_patrol.py`
当前最小真实执行脚本。

作用：
- 抓取部分 Reddit RSS 热门内容
- 抓取部分 GitHub topic 页面中的候选仓库链接
- 把结果写入本地文本文件，作为一次真实巡逻的原始输出

当前抓取范围：
- Reddit：
  - `r/LocalLLaMA`
  - `r/OpenAI`
  - `r/ChatGPT`
- GitHub Topics：
  - `ai-agents`
  - `browser-automation`

当前输出文件：
- `patrol-run-2026-04-17.txt`

## 当前工作流

### 总工作流
1. 在真实工作里发现卡点、瓶颈或不稳定点
2. 判断能否当场补洞，而不是只临时绕过
3. 需要外部输入时，再用 Reddit / GitHub 巡逻找信号、找实现、找方法
4. 把解决结果沉淀成脚本、skill、模板、执行顺序或文档
5. 记录本次升级点，确保下次更稳、更快、更强

### 当前执行方式
目前主要是 **文档定义 + 脚本补洞 + 外部巡逻 + 人工判断** 的组合：
- 文档负责定原则和边界
- 脚本负责补实际工作中的能力缺口
- 外部巡逻负责寻找新方法和候选实现
- 人负责筛选、判断、收敛和回灌

所以当前阶段仍是：

**与实际工作绑定的半自动自我进化原型，而不是全自动研究系统。**

- Windows
- Python 3.x
- 能正常访问 Reddit / GitHub 页面

如果要继续往前做，优先级建议如下：
