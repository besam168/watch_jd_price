# Codex CLI Exec Completion Workflow

> 用途：当用 `codex exec` 做最小真测或自动化验证时，避免把“已经打印出成功文本”误当成“进程已经干净退出、结果可以直接对外下结论”。

---

## 适用场景

当出现以下情况时，优先走这份 workflow：
- 用 `codex exec` 做最小路由真测
- 用 `codex exec` 跑一次性短任务
- tool 返回了目标输出，但进程还显示 `still running`
- 后续又看到 `signal SIGKILL`、超时或会话未正常收尾

一句话原则：

**Codex CLI 里“模型回答成功”不等于“进程生命周期成功结束”。**

---

## 默认判断顺序（写死）

### 1. 先看有没有拿到核心任务输出
例如目标是：
- `Reply with exactly: CODEX_ROUTE_OK`

如果日志里已经明确出现：
- `CODEX_ROUTE_OK`

说明：
- 模型调用本身大概率已经成功
- 路由、key、模型名至少在这次请求层面是通的

### 2. 但不要立刻把它当“整条 exec 已干净成功”
如果 tool 侧还显示：
- `Process still running`
- 或之后出现超时 / `SIGKILL`

那就要把结果拆成两层表述：

#### 请求层成功
- 模型已经回了目标文本
- 说明 provider / model / key / request 这层是通的

#### 进程层未完全收口
- Codex CLI 进程没有在期望时间内自然退出
- 不能说“整个 exec 生命周期完全健康”

### 3. 对外表述时必须分层
正确说法应类似：
- “最小真测已拿到目标文本，说明模型路由已可用；但本次 `codex exec` 进程没有及时干净退出，进程层收口仍需单独看。”

不要直接把它说成：
- “全部完全正常”

### 4. 后续动作
如果后面还要把 Codex CLI 接进自动化流程，需要继续确认：
- 是否存在 CLI 本身退出延迟
- 是否是宿主进程 / tool session 超时策略导致被杀
- 是否需要缩短输出、降低 reasoning、或改用更短的非交互调用方式

---

## 推荐实操模板

### 目标
执行：
- `codex exec "Reply with exactly: CODEX_ROUTE_OK"`

### 观察结果 A
日志里出现：
- `CODEX_ROUTE_OK`

### 观察结果 B
但 tool 仍显示：
- `Process still running`
- 随后系统消息出现 `SIGKILL`

### 正确结论
- **请求层：成功**
- **进程层：未完全收口**

---

## 常见坑

### 坑 1：看到目标文本就马上宣布“链路完全没问题”
这会忽略一个事实：
- CLI 可能还挂着
- session 可能没正常结束
- 自动化场景里这会影响稳定性

### 坑 2：看到后续 `SIGKILL` 就反过来宣布“前面测试失败了”
也不对。

如果目标文本已经拿到了，说明：
- request 层是通的

只是：
- process lifecycle 还不够干净

### 坑 3：不区分“请求成功”和“进程成功”
这是最容易让验收口径混乱的点。

默认处理：
- 所有 Codex CLI 验收都分成这两层写

---

## 最小检查表

- [ ] 是否已经拿到目标输出文本
- [ ] 进程是否已自然退出
- [ ] 是否出现 timeout / SIGKILL / still running
- [ ] 对外结论是否区分了“请求层成功”和“进程层成功”

---

## 一句话总结

**`codex exec` 的正确验收口径是两层：先看模型有没有回到目标结果，再看进程有没有干净退出；两层都过，才算完整成功。**
