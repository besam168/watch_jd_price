# 搜索 / 抓取不稳时的备用工作流

> 用途：当真实工作里需要查资料、抓网页、做外部信息获取时，如果主路线不稳定，就不要临场乱切。按这份固定顺序执行，目标是：**先把事做成，再把能力补上。**

---

## 适用场景

当出现以下任一情况时，启动这份工作流：
- Firecrawl 不稳
- 内置搜索不稳
- 某个网页抓取结果不稳定
- 抓取路线今天能用、明天失效
- 同一类任务每次都靠临场换方案
- 外部信息获取已经成为工作流瓶颈

---

## 总原则

1. **先保证任务完成，再谈优雅**
2. **不要只押一个抓取源**
3. **不要只靠临场记忆切换路线**
4. **每次 fallback 都要尽量沉淀成可复用资产**

一句话：

**主路线不稳时，不是“今天先这样”，而是进入固定 fallback 顺序。**

---

## 初版执行顺序（V1）

### Level 0：先判断任务类型
先把当前任务分清楚，否则后面 fallback 会乱。

#### A. 只是“找答案 / 查一个事实”
优先考虑：
- 现成搜索
- 专用 API
- 轻量网页抓取

#### B. 需要“读完整网页内容”
优先考虑：
- 网页抓取 / scrape
- 网页提取 / markdown
- 本机 fallback 抓文本

#### C. 需要“跨多个站点做观察 / 比较 / 研究”
优先考虑：
- `self-evolution-radar`
- 结构化摘要
- 回写 worklog

---

### Level 1：主路线先试一次
如果任务本来已有默认主路线，先跑一次，但不要无限重试。

例如：
- 先用首选搜索
- 先用首选抓取方案
- 先用已有 skill

**规则：**
- 最多快速尝试 1 次
- 失败就进入下一层
- 不要在不稳路线里来回死磕

---

### Level 2：切到轻量替代路线
如果主路线不稳，优先切更轻的方案。

#### 搜索类 fallback
优先顺序：
1. 专用 API（如果有明确结构化工具）
2. 内置搜索 / 轻量网页抓取
3. 直接抓目标网页内容
4. 必要时手工指定站点缩小范围

#### 单页抓取类 fallback
优先顺序：
1. 轻量网页提取
2. 直接抓目标 URL
3. 本机脚本读取页面内容
4. 必要时改成“只抓关键页面，不追求全量”

**目标：**
先拿到能工作的最小结果，不追求一步到位最优雅。

---

### Level 3：切到本机可控路线
如果在线路线持续不稳，就优先考虑本机可控方案。

例如：
- 本机 Python 脚本
- 本机 PowerShell 脚本
- 已有本地 skill
- 直接访问目标 URL 并自己提取关键内容

这一步的核心不是“更高级”，而是：
- 更可控
- 更容易调试
- 更容易固定成下次复用流程

---

### Level 4：如果问题已成能力缺口，进入 self-evolution 闭环
如果当前问题已经符合下面任一条，就不要只把这次任务做完：
- 同类问题重复出现
- 每次都靠临场换路线
- 当前工作流明显缺一条备用路径
- 外部信息获取已经成瓶颈

此时进入：

`worklog -> watchlist/topic 判断 -> self-evolution-radar -> summary -> 回写 worklog`

也就是：
1. 先记 `worklog`
2. 判断 topic
3. 跑一轮 radar
4. 看 summary
5. 把结论回写到 worklog
6. 明确应沉淀成什么资产

---

## 回写时必须明确的 4 件事

每次走完 fallback，至少要回答：
1. 这次哪个环节不稳
2. 这次最后靠什么方式做成
3. 下次默认应该先走哪条备用路径
4. 这件事应不应该沉淀成脚本 / skill / 模板 / SOP

---

## 当前建议的默认 fallback 资产方向

基于当前这轮真实样板，后续最值得继续补的资产是：

### 1. 备用抓取路线清单
把不同任务类型对应的备用路线列清楚。

### 2. 本机脚本 fallback
把那些反复会用到的“最小可用抓取动作”沉淀成本机脚本。

### 3. 可复用巡逻脚本
对“需要跨站点看趋势 / 看方法 / 看实现”的任务，优先走 radar，而不是临场乱搜。

### 4. topic / watchlist 外置配置
减少每次改 Python 才能换方向的维护成本。

---

## 一句话版本

**搜索 / 抓取不稳时，先试主路线一次；不稳就切轻量替代；再不稳就切本机可控路线；如果已经是重复性能力缺口，就进入 self-evolution 闭环。**
